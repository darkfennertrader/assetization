## Decisions made

### 1. Demo integration model: top-level redirect (no iframe, no proxy)

Overwatch (and all future demo apps) will be **independently deployed ACA apps**,
each with their own Entra ID App Registration and their own in-app group check.
The showroom catalog redirects the browser to the demo app's URL
(`window.location.href = demo.launchUrl`). Silent SSO is provided by the shared
Entra ID session cookie. No BFF proxy, no iframe embed in phase 1.

Owner: Miki (developer) proposed and confirmed this model. Architecture updated
in `showroom/phase1/showroom-phase1-internal.md` §4.

### 2. Group-based access: shared Entra ID group is the access guarantee

Both the showroom and every demo app enforce the same `FinCrime-Showroom` Entra
ID security group. Any user who passes the showroom gate will silently pass each
demo app's gate. Catalog visibility (which tiles appear) is a second filter
controlled by `demo-map.json` committed to the repo.

Owner: agreed by all three.

### 3. Authorization enforcement: in the app, from token claims

For phase 1, group membership is checked in the Next.js API route reading the
`groups` claim from the Entra ID token. No platform-layer enforcement (no APIM
`validate-jwt` policy). APIM is a phase-2 addition.

Owner: Aditya proposed in-app for phase 1; Raimondo agreed.

### 4. IaC choice: developer's preference

Bicep or Terraform — Raimondo (architect) left the choice to the implementer.
No preference from the team.

### 5. CI/CD: Azure DevOps pipeline on `main`

Two pipelines: one for the showroom app, one for Overwatch. Both follow the same
pattern: push to `main` → build Docker image → push to ACR → update ACA app.
Workload Identity Federation service connection (no client secret to rotate).

Owner: Aditya provisions the pipelines. Miki pushes code once repo access is
confirmed.
