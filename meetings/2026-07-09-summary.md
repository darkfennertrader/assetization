## Overwatch Demo

Overwatch is a **transaction-monitoring investigation demo** for banks, not a product or MVP.
It started as a notebook demo to show Peter's team that complex agentic UX could be built in two days.
It then grew organically: Azure deployment, multi-user, access control -- all layered on as afterthoughts.

What works: the chat (assistant answers investigation questions from JSON data), location map queries.
What is mocked: most other panels and data sources.
The underlying idea: investigators spend 75% of their time copying and pasting across systems.
Overwatch reimagines the full investigation workflow with agentic AI, using Databricks in production
but JSON for the demo. The goal is to show prospective clients what is possible, not to sell a product.

## Marketplace -- Peter's Reframing

The word "marketplace" was accepted but Peter's definition differs from a runtime service catalogue.

Peter's vision: a **library / warehouse** of pre-built things -- blueprints, IaC templates, proposal
PowerPoints, code snippets, presentation decks -- that a PwC employee can search when starting a
new engagement so they do not reinvent everything from scratch. An agent layer on top helps match
what you describe to what already exists.

**It is not** an MCP server or a runtime API catalogue that developers call at runtime.
MCP exposure of reusable capabilities was discussed but Peter explicitly deferred it:
PwC is a complicated organisation; even building the capability does not mean it will be approved.

Action: clarify internally what the exact expectation is before designing a technical solution.

## Showroom -- Decisions Made

Peter confirmed the following decisions, effective immediately:

### Authentication for external prospects

Use **Microsoft OAuth and Google OAuth** (consumer accounts, no account creation).
Prospects open a single public showroom URL, click "Sign in with Microsoft" or "Sign in with Google",
complete a standard PKCE flow, and are in.
No QR codes, no Entra External ID account-creation ceremony, no self-signed JWT session tokens.

### QR code -- dropped

The QR-code feature was originally proposed by Miki in a conversation with Adrian.
Peter's assessment: development effort of days, ongoing maintenance, no clear added value over
the OAuth login that requires zero configuration on the user side.
Decision: drop from Phase 2 entirely. Leave the door open to revisit only if OAuth proves
insufficient for specific demo-event scenarios.

### Time-based access and per-user usage tracking (new layer)

Based on how Peter's previous showroom worked, every external user's session must be tracked:

- Which user logged in and when (first seen, last seen, total connections).
- Which demos they opened and when.
- How many requests they generated in each demo.
- A per-user message quota, configurable by the presenter, top-up-able on request.
- (Legal review required before logging message content itself.)

This tracking lives in the showroom BFF and Cosmos DB, not inside each demo app.
Demo apps report their per-session message count back to the showroom via a lightweight callback.

### Access control model

Presenter Admin (PwC salesperson, authenticated via Entra ID) manages an email allow-list:

- Add or remove external user emails.
- Set which demos each user may see (`allowedDemoIds`).
- Set and top up the user's message quota.
- View live analytics: last login, demos opened, request counts.

A prospect not on the allow-list sees "You are not authorised -- contact your PwC host."

### Deployment

Azure Container Apps (scale-to-zero, near-zero idle cost) confirmed.
Azure Function keep-warm every 5 minutes during office hours to avoid cold-start appearing as
an outage to clients.
APIM deferred to Phase 3 -- Phase 2 uses a NextAuth-only stack to reduce cost and complexity.
